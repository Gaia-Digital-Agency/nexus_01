# Technical Audit — ypi-asia.com

> **In plain terms (for the team):** Your site is generally crawlable and secure, with Google able to access your content (sitemap is declared, redirects are good) and strong security headers in place (HTTPS enforced, X-Frame-Options, X-Content-Type-Options, X-XSS-Protection). However, the single biggest risk is a significant security vulnerability: your WordPress database uses the default table prefix (`wp_`), making it much easier for attackers to target your database.

**Server:** ce01 · **Audited:** 2026-06-11 · **Status:** live

## Verified signals
- HTTP/redirects: `http://ypi-asia.com` redirects via 301 to `https://www.ypi-asia.com/`.
- robots.txt: Disallows `/wp-admin/` but allows `/wp-admin/admin-ajax.php`.
- sitemap: `https://www.ypi-asia.com/sitemap_index.xml` is declared and accessible (HTTP 200). The sitemap index is generated by Rank Math SEO Plugin and includes multiple sub-sitemaps.
- HTTPS/headers: HTTPS is enforced via 301 redirect. The server is `nginx/1.24.0 (Ubuntu)`. Security headers `X-Frame-Options: SAMEORIGIN`, `X-Content-Type-Options: nosniff`, and `X-XSS-Protection: 1; mode=block` are present.
- Platform/version: WordPress.
- Plugins/theme/security: `WP_DEBUG` is set to `false`. The site uses 20 plugins: `adminify`, `advanced-custom-fields-pro`, `creame-whatsapp-me`, `custom-post-type-ui`, `elementor`, `elementor-pro`, `error-log-monitor`, `forminator`, `google-site-kit`, `hostinger`, `hostinger-ai-assistant`, `hostinger-easy-onboarding`, `hostinger-reach`, `instagram-feed`, `litespeed-cache`, `seo-by-rank-math`, `translatepress-multilingual`, `translatepress-personal`, `wp-mail-smtp`, `wp-mail-smtp-pro`. JSON-LD schema is present on the homepage.

## Technical findings
| Area | Finding | Severity |
|---|---|---|
| Security | Default WordPress database table prefix `wp_` is used. | High |
| Performance/Maintainability | Large number of active plugins (20) increases potential for conflicts, bloat, and security vulnerabilities. | Med |
| Security | WordPress REST API endpoints are exposed (`/wp-json/`). | Low |
| Security | No `generator` meta tag is present, which is good for obscuring CMS version. | Info |
| Performance | `litespeed-cache` plugin is active, indicating a caching solution is in place. | Info |
| Monitoring | `error-log-monitor` plugin is active, which is useful for identifying issues. | Info |

## Could not verify
- CMS/platform specific version (e.g., WordPress 6.x.x)
- Plugin/theme updates status
- Plugin/theme conflicts
- Exposed files (beyond what's implied by robots.txt)
- Authentication security (e.g., strong passwords, 2FA)
- Speed/Core Web Vitals signals
- Overall indexability status (beyond robots.txt and sitemap presence)
- Google Search Console verification status

## Top technical fixes (analysis only — NOT executed)
1.  Change default WordPress database table prefix `wp_` to a unique, random string to enhance database security.
2.  Review and audit active plugins, removing any unnecessary or redundant ones to reduce bloat, improve performance, and minimize potential security vulnerabilities.
3.  Consider restricting access to or disabling the WordPress REST API if not actively used by external applications, to reduce potential attack surface.
